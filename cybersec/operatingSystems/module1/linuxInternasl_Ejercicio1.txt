1️⃣ Arquitectura Linux (Cómo piensa el sistema)
🔥 Vista real
4
📌 Estructura real

Linux no es solo “un OS”.

Se divide en:

Hardware
  ↑
Kernel
  ↑
System Calls
  ↑
Userland (bash, apps, servicios)

🧠 Qué es el Kernel

El kernel es:

Administrador de memoria

Scheduler de procesos

Controlador de dispositivos

Gestor de permisos

Implementador de syscalls

Cuando haces:

cat file.txt


No es cat quien accede al disco.
Es el kernel mediante una syscall (read()).

🧨 Por qué importa para hacking

Porque:

No puedes escalar privilegios solo en userland

Toda escalación real implica abusar del kernel

Los exploits de verdad explotan errores en:

syscalls

drivers

memory management

race conditions

2️⃣ Userland vs Kernel (Donde vive el poder real)
Capa	Nivel	Qué puede hacer
Userland	Bajo	Ejecutar programas
Kernel	Root absoluto	Control total del sistema
🔥 Ataques comunes aquí

4️⃣ Permisos reales (rwx ≠ seguridad)

Todos creen que Linux es seguro por:

-rwxr-xr-x


Pero eso es superficial.

🔥 Lo que realmente importa
SUID bit
chmod u+s file


Ejemplo:

-rwsr-xr-x  root  root  /usr/bin/passwd


Eso significa:

Se ejecuta como root.

Si puedes inyectar algo ahí → escalación.

🔥 Sticky bit

Importante en /tmp.

🔥 ACLs

Permisos extendidos.

🧪 EJERCICIO 1 — Observar la frontera Userland → Kernel
🎯 Objetivo

Entender qué es una syscall en acción.

Paso 1 — Instala strace
sudo apt install strace

Paso 2 — Ejecuta algo simple
strace ls


Ahora responde:

¿Qué syscalls ves?

¿Qué ocurre antes de que se liste el directorio?

¿Ves open(), read(), write()?

🧠 Reflexión técnica

Cuando ejecutas:

ls


El flujo real es:

ELF loader → libc → syscall → kernel → VFS → driver → disk


Tu misión:

Ejecuta:

strace -f -e trace=process ls


¿Ves fork()?
¿execve()?

Ahí está el límite userland/kernel.

🧪 EJERCICIO 2 — Anatomía de un Proceso
🎯 Objetivo

Entender cómo Linux representa un proceso internamente.

1️⃣ Ejecuta un proceso en segundo plano

2️⃣ Explora su representación en /proc

Archivos críticos:

status
maps

mem
cmdline
environ

🔥 Analiza memoria del proceso
cat /proc/<PID>/maps


Responde:

¿Dónde está el stack?

¿Dónde está el heap?

¿Dónde están las librerías?

Esto es crucial para:

ASLR

Exploit development

Memory corruption attacks

🧪 EJERCICIO 3 — Entender el ELF Loader
🎯 Objetivo

Ver cómo Linux ejecuta binarios realmente.

1️⃣ Analiza un binario
file /bin/ls


Verás que es:

ELF 64-bit LSB executable

2️⃣ Usa ldd
ldd /bin/ls


Esto muestra:

Librerías dinámicas

Dependencias

3️⃣ Usa readelf
readelf -a /bin/ls


Busca:

Entry point

Sections

Program headers

🧠 Entiende esto:

Cuando ejecutas un binario:

Kernel valida ELF header

Carga segmentos en memoria

Mapea libc

Salta al entry point

Ahí comienza todo.

3️⃣ Observa ASLR

🧪 EJERCICIO 5 — Namespaces reales
🎯 Objetivo

Entender cómo Docker realmente aísla procesos.

1️⃣ Ver namespaces actuales

2️⃣ Crea uno manualmente

¿Ves todos los procesos del sistema?

No.

Acabas de crear un PID namespace.

Eso es la base de contenedores.

🧪 EJERCICIO 6 — Capabilities en acción
