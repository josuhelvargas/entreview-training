3) Query Pipeline interno: de Expression Tree a SQL

Cuando escribes:

var q = ctx.Users
  .Where(u => u.Age > 18)
  .Select(u => new { u.Id, u.Name });


Eso es IQueryable: no corre nada aún. Solo crea un Expression Tree.

3.1 Etapas internas (vista de alto nivel)

Construcción del Expression Tree (C# + LINQ)

Preprocesamiento (normaliza, expande includes, aplica filtros globales, etc.)

Traducción (Expression → “árbol SQL interno”)

Postprocesamiento (optimizaciones, simplificaciones, parámetros)

Compilación (se crea un “plan ejecutable” que produce resultados)

Ejecución (DbCommand → DbDataReader)

Materialización (rows → objetos / proyecciones)

Tracking opcional (si aplica)

¿Cómo “traduce” realmente?

EF Core usa visitors (ExpressionVisitor) para caminar el árbol.

Cuando encuentra algo traducible (comparaciones, funciones soportadas), lo convierte a nodos relacionales.

Si encuentra algo NO traducible:

En EF Core 2.x: podía intentar client-eval silencioso.

En EF Core 3+: error a menos que tú cruces el boundary (AsEnumerable()).

4) ¿Qué es QueryCompilationContext?

Es el “contenedor de compilación” para una query:
el lugar donde EF Core junta todo lo necesario para transformar un Expression Tree en un plan ejecutable.

Incluye cosas como:

El modelo (IModel)

El provider (SQL Server, etc.)

Opciones (tracking, split queries, timeouts, etc.)

Logger/diagnósticos

Parámetros y cómo se van a manejar

Si se usará buffering, streaming, etc.

Piensa: “contexto de compilación = settings + servicios + provider + modelo” para producir un “delegate” de ejecución.

5) Cache interna de queries: ¿cómo funciona?

EF Core cachea principalmente:

El plan compilado de una query “con la misma forma” (misma estructura de Expression Tree).

Separando valores como parámetros (parameterization).

Ejemplo:

ctx.Users.Where(u => u.Id == id)


La forma es la misma aunque id cambie, entonces:

EF compila una vez el plan

Reusa el plan

Cambia solo el parámetro

Clave mental: EF Core cachea “shape”, no resultados.

6) IQueryable vs IEnumerable en el pipeline

IQueryable: “esto todavía se puede traducir a SQL”

Todo lo que hagas aquí puede ir al server.

IEnumerable: “ya estoy en memoria”

Todo lo que hagas aquí ocurre en el cliente.

El boundary real ocurre cuando:

Llamas ToList(), First(), Count(), etc. (materializa)

O fuerzas AsEnumerable() / ToList() temprano

La regla:

Mantén IQueryable hasta el último momento si te importa performance.

7) Change Tracker: Snapshot vs Notification-based
7.1 Snapshot tracking (default común)

EF guarda un “snapshot” del estado original.
Al guardar, compara “original vs actual”.

Pros:

No requiere que tus entidades implementen nada especial.
Contras:

Comparación cuesta (aunque EF Core lo ha optimizado mucho).

7.2 Notification-based tracking

Si tus entidades implementan notificaciones (como INotifyPropertyChanged), EF puede reaccionar a cambios sin comparar todo.

Pros:

Menos costo de DetectChanges.
Contras:

Más complejidad en entidades.

8) DetectChanges internamente y AutoDetectChangesEnabled
DetectChanges

Recorre entidades trackeadas y marca estados:

Unchanged, Modified, Added, Deleted

Detecta cambios por property

Lo ejecuta:

Antes de SaveChanges() (si está habilitado)

En algunos escenarios de tracking

AutoDetectChangesEnabled = false

Útil en cargas masivas (bulk insert con AddRange):

Evitas llamar DetectChanges por cada entidad

Luego llamas una vez al final (o dejas que SaveChanges lo haga)

Tradeoff: si lo apagas, tú te haces responsable de no dejar estados inconsistentes.

9) AsNoTrackingWithIdentityResolution: qué hace

AsNoTracking():

No tracking: más rápido, menos memoria

Pero si el query produce el mismo entity varias veces (joins), podrías obtener objetos duplicados.

AsNoTrackingWithIdentityResolution():

Sigue siendo no tracking (no se queda en ChangeTracker)

Pero usa un “identity map” temporal durante materialización

Resultado: misma entidad = misma instancia dentro del resultado.

Útil cuando:

Tienes graph con joins e includes

No quieres tracking pero sí consistencia de referencias

10) Performance: CompiledQuery vs CompiledModel
CompiledQuery

Compila la ejecución de una query específica:

Menos overhead por compilación repetida

Útil en endpoints súper calientes (alta frecuencia)

Úsalo cuando:

Query se ejecuta muchísimo

Forma estable

Performance critical

CompiledModel

Precompila el modelo (mapping, metadata, conventions)

Reduce costo de startup / primera query

Excelente para microservicios con cold start o serverless

Qué overhead elimina:

Construcción del modelo en runtime

Aplicación de conventions

Descubrimiento de mappings por reflexión (gran parte)

11) “Demasiadas Includes”: qué pasa realmente

Include puede provocar:

Joins enormes (cartesian explosion)

Duplicación de rows (el reader devuelve filas repetidas para reconstruir graph)

Mucha memoria y CPU en materialización

Mitigaciones típicas:

Proyecciones (Select) en vez de Include cuando solo necesitas DTO

AsSplitQuery() (divide en varias queries para evitar explosion)

Cargar por partes según caso (explicit loading o queries específicas)

12) Arquitectura: Repository vs EF Core
¿EF Core viola Repository?

No “viola”, pero en la práctica:

DbSet<TEntity> ya funciona como un repositorio.

DbContext ya es Unit of Work.

Crear repositorios genéricos encima de EF a veces:

Solo agrega ruido

Oculta IQueryable y mata composabilidad

Complica performance

Repository sí tiene sentido cuando:

Tu dominio exige invariantes y métodos específicos

Encapsulas queries complejas y consistentes

Evitas que toda la app haga LINQ libre sin control

13) DbContext como Unit of Work y Scoped DI

Sí: DbContext es Unit of Work (trackea cambios + SaveChanges).
En ASP.NET Core lo correcto es Scoped:

1 request = 1 scope = 1 DbContext (generalmente)

No debe ser Singleton.

14) Alta concurrencia: qué pasa

Problemas comunes:

Dos requests actualizan el mismo registro

Sin control, “last write wins”

Solución típica: optimistic concurrency

rowversion / timestamp

Concurrency tokens en EF

EF detecta conflicto y lanza DbUpdateConcurrencyException

Además:

Manejo correcto de transacciones

Retries (en escenarios transient, no en lógica de conflicto real)

15) Bases modernas: JSON, Value Converters, Provider Model, Cosmos
JSON mapping (EF8)

Algunas bases soportan columnas JSON con queries dentro del JSON.

EF permite mapear partes del JSON a propiedades/owned types (depende del provider).

Value Converters

Transforman valores entre:

Tipo del dominio (ej. Money, Email, StrongId)

Tipo persistible (decimal/string/guid)

Sirven para:

Mantener dominio limpio

Persistir sin ensuciar entidades

Provider model

EF Core define “abstracciones”
y el provider implementa:

Cómo generar SQL (dialecto)

Qué funciones soporta

Tipos, parámetros, limitaciones

Translators específicos

Cosmos sin SQL real

EF Core usa un provider que:

Traduce LINQ a la “forma de consulta” de Cosmos

Maneja particiones y limitaciones propias
