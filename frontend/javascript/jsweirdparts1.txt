Single-threaded + Event-driven

JavaScript engine es single-threaded.

Solo una cosa puede ejecutarse en el call stack a la vez.

Pero…

El runtime es concurrente.

¿Cómo funciona realmente?



Componentes reales

Call Stack (engine)
Web APIs (browser C++)
Task Queue (macrotasks)
Microtask Queue
Event Loop

Flujo real

1.JS llama setTimeout
2.setTimeout NO vive en JS
3.Vive en Web APIs (C++)
4.Cuando termina → callback a Task Queue
5.Event Loop verifica stack vacío
6.Mueve callback al stack





Procesos en Chrome

Chrome es multi-proceso.
Browser Process
Renderer Process
GPU Process
Network Process


¿Dónde vive JavaScript?
En el Renderer Process.






Quién implementa Web APIs?
En Chrome:

Rendering Engine: Blink
JS Engine: V8

Las Web APIs como:
setTimeout
DOM
fetch
addEventListener
Son implementadas en Blink (C++).


V8 se comunica con Blink mediante bindings. <--!
Flujo real DOM + JS
JS llama document.getElementById
V8 delega a Blink
Blink accede al DOM
Resultado vuelve a V8




Render Pipeline (para entender la integración real)
Cuando JS modifica el DOM:
1.DOM tree update
2.Style recalculation
3.Layout
4.Paint
5.Composite
Si JS bloquea el thread, el render se detiene.
